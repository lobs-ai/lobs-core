## Executive Summary

Task `de4a2c9a` was a **legitimate phantom completion**: a 41-second, 1-turn agent run that declared `succeeded=true` while producing zero output artifacts. The agent described its plan in Turn 1 and immediately emitted `end_turn` without executing any tools.

Task `44b653d0` (the investigation of `de4a2c9a`) was a **false positive phantom detection**: a 767-second, 15-turn, read-only investigation that legitimately produced no file changes or commits — but was still flagged as `needs_review` because the post-success validator cannot distinguish read-only analysis tasks from broken ones.

Both tasks have now been fully root-caused. The underlying issues are distinct and require separate mitigations.

## Root Cause Analysis

### Task de4a2c9a — Primary Phantom (Genuine Bug)

**Root cause: Local model produced a reasoning trace but emitted `end_turn` without executing any tools.**

Evidence from session `967527d77c8b7134.md`:
- 1 turn total, 41s elapsed
- Agent summary: *"I need to find and update active tasks to include their tier and completion percentage. Let me search for active tasks and their current descriptions."*
- No tool calls made: 0 execs, 0 reads, 0 writes, 0 memory ops
- The model (`lmstudio/qwen/qwen3.5-9b`) completed its reasoning chain (`<think>...</think>`) and then — likely due to a token budget constraint or output formatting failure — emitted `end_turn` rather than issuing its first tool call

**Contributing factors:**
1. **Small local model (9B params):** Qwen 3.5-9B frequently runs long chain-of-thought reasoning traces, leaving few tokens for actual tool dispatch
2. **Task was redundant:** The work (adding tiers/status to task descriptions) had already been completed via commit `35dad6c` and documented in `lm-studio-tasks-restored.md`. The model may have correctly reasoned "this is already done" and then incorrectly produced `end_turn` instead of a summary or verification step
3. **No minimum-tool-call guard:** The control loop does not enforce that a worker run must make at least one tool call before being eligible for auto-close as `succeeded`

**Disposition:** Task `de4a2c9a` was **redundant** — the active task descriptions already included tiers and status markers. The phantom completion did not cause any missing work.

### Task 44b653d0 — Secondary Phantom (False Positive in Validator)

**Root cause: The `post-success-validator` cannot distinguish read-only investigation tasks from phantom completions.**

Evidence from session `b10e9007dc1813cd.md`:
- 15 turns, 767s elapsed
- Tool calls made: `exec` ×7, `memory_read` ×7, `memory_search` ×4, `read` ×3 = **21 tool calls total**
- 0 file writes, 0 edits, 0 git commits — all tools were read-only
- The agent completed a thorough investigation and produced a detailed written conclusion in Turn 15
- The DB record (`worker_runs.summary`) contains the full analysis text — it was recorded

**Why it was flagged:**
The `post-success-validator` checks:
1. `commit_shas` — empty (no git commits)
2. `files_modified` — empty (no writes/edits)
3. File mtime scan of `repo_path` — no files changed

A pure investigation task will fail all three checks by design. The validator correctly detected "no filesystem artifacts" but incorrectly inferred "phantom completion."

**The agent did produce output** — it was written into:
- `worker_runs.summary` (DB field, full analysis in 15 turns)
- The session markdown file `b10e9007dc1813cd.md` (346KB JSONL, 15 turns of tool use)

The validator does not check these locations.

## Recommendations

### Immediate (this session)

1. **Close task `de4a2c9a`** with status `done`, note: "Redundant — work completed in commit 35dad6c. Agent ran but correctly found nothing to update. Local model (9B) emitted end_turn without tool dispatch (token budget exhaustion pattern)."

2. **Close task `44b653d0`** with status `done`, note: "Investigation complete. RCA published at docs/post-mortems/2026-03-23-phantom-task-44b653d0.md. False positive phantom detection — investigation task with no commit is valid."

### Short-term (within current sprint)

3. **Add minimum tool-call guard in control loop:**  
   If `succeeded=true` AND `turns=1` AND `tool_calls=0`, auto-set `validity_result=likely_phantom` without waiting for file scan.  
   File: `lobs-core/src/orchestrator/control-loop.ts` around the `validatePostSuccessArtifacts` call path.

4. **Add `task_nature` classification to validator:**  
   Tasks whose titles begin with "Investigate", "Analyze", "Research", "Audit", or "Review" should have a relaxed artifact requirement. The validator should require at least one of: commit, write, OR a `summary` field with >200 chars. Investigation tasks that produce a full session summary pass the last criterion.

5. **Improve validator to check `worker_runs.summary` length:**  
   A summary >500 characters is strong evidence the agent did real work, even without a commit.

### Long-term

6. **Consider adding `expected_output_type` field to tasks:**  
   Values: `code_change` (requires commit), `investigation` (summary sufficient), `documentation` (markdown file required). This makes artifact validation deterministic per task type.

7. **Local model token budget tuning:**  
   For tasks assigned to `lmstudio/qwen/qwen3.5-9b`, reduce system prompt length or add an explicit "make at least one tool call before summarizing" instruction to combat the reasoning-but-no-dispatch pattern.
